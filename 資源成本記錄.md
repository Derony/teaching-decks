### Context 重量級項目（經驗法則，每項 ≈ token）

- 看一張圖（Read 圖檔／截圖驗收）≈ 1,000–1,600
- 載入一個 soil 技能 ≈ 3,000–4,000
- 寫一份完整 HTML 模板（~20KB）≈ 8,000（輸出也算）
- 一般指令往返 ≈ 300–800
- **教訓**：圖片驗收是大戶——一份簡報生 3 圖＋逐頁截圖 5 次 ≈ 12K tokens 起跳。能用 eval 量測代替截圖就用 eval（量測一次 ≈ 200，截圖一次 ≈ 1,500）

| 項目 | 數據 |
|---|---|
| 工時 | **約 110 分鐘**（讀解 PPT 25 分 → 畫格管線 40 分 → 組裝 HTML 20 分 → 驗收修 2 bug 25 分） |
| 生圖 API | **0 次／NT$0**（全用原 PPT 素材） |
| Context 估算 | ≈ 70–90K tokens（大戶：原 PPT 渲染圖 Read 8 張＋畫格抽查＋驗收截圖 6 張） |

**教訓**：
1. **翻新前先渲染每頁＋解析動畫 XML 再動工**——這次因為先用文字萃取就重設計，方向錯誤重做一次，多花近 2 小時。正確順序省一半時間
2. 復刻型簡報生圖費為零，成本主要在 context（讀圖驗收）與工時
3. PS 5.1 跑含中文的 .ps1 必須 UTF-8 **BOM**

#### #1c 同簡報・結構化可編輯復刻版（2026-06-11 17:49–18:57）

`slides.html` 整頁圖片化、難二次編輯；本版**逐物件還原**原 PPT 成真 HTML 元素
（文字保留文字、圖片獨立、形狀向量化、Ink 取 fallback 圖），原始碼結構化、好二次優化。

| 項目 | 數據 |
|---|---|
| 工時 | **約 70 分鐘**（17:49 首支腳本 → 18:39 成品定稿 → 18:57 存檔；另含開工後「可編輯定義」討論＋PPT 勘查約 20 分） |
| 生圖 API | **0 次／NT$0**（全用原 PPT 既有 media，未呼叫 draw） |
| 工具呼叫 | ≈ 70–80 次（含 preview 截圖/eval 驗收約 20 次） |
| Context 估算 | ≈ 90–120K tokens：寫 2 支腳本（多輪 Edit）＋截圖 6＋原圖對照 Read 3＋XML 勘查 dump 5 |
| 實際 context／time limit 變化 | （使用者未提供，待校正） |
| 方法 | spTree 物件幾何/文字/字型/**行距 lnSpc**/圖片/形狀 → layout.json；設計畫布 1440×810（EMU÷12700）＋CSS scale；cxn/ellipse→SVG、Ink→`mc:Fallback` 光柵圖；圖片**去重**＋壓縮 15.6MB→2.57MB |

**教訓**：
1. **「可編輯」先對齊定義**——使用者要的是「復刻不圖片化、保留結構、好二次優化」，不是網頁即時改字；我先誤解多做了 `contenteditable`。動工前一兩輪確認需求省返工
2. 結構化復刻關鍵在 PPTX 結構認知：「Freeform」多是 blipFill 圖、Ink 有 `mc:Fallback` 光柵圖、標題偏位是因 `lnSpc` 大行距——先勘查 XML 少走彎路
3. 單檔肥因是「跨頁重複內嵌」而非圖片本身大（media 僅 2.3MB）；**去重（JS 字典）比壓縮更有效**
4. 全程 eval 量測代替部分截圖（掃 8 頁物件統計一次 eval 抵 8 次截圖），省 context
